## 1. 專案背景

AIDC.work 是 Jack Y. J. Hwang 的企業 AI 發展顧問網站，主要呈現 AI 工具培訓、Vibe Coding、AI 系統開發、AI Agent 工作流與企業 AI 導入路線圖等服務。

目前專案已具備靜態內容頁、Markdown / MDX 內容結構、預約頁、預約資料庫模型、管理員登入、後台預約管理與時段管理雛形。下一階段需將網站從「顧問網站與預約入口」擴展為「可管理會員、內容、權限與營運流程的顧問資產平台」。

## 5. RBAC 權限架構

### 5.1 核心概念

RBAC 採用以下關係：

```text
User → UserRole → Role → RolePermission → Permission
Menu → Permission
Route / API → Permission
```

每位後台用戶可擁有一個或多個角色，每個角色可綁定多個權限。後台選單、頁面路由與 API 操作需透過權限代碼進行控管。

### 5.2 權限代碼命名規則

權限代碼建議採用：

```text
resource.action
```

範例：

| 權限代碼 | 說明 |
|----------|------|
| `adminUser.read` | 查看後台用戶 |
| `adminUser.create` | 新增後台用戶 |
| `adminUser.update` | 編輯後台用戶 |
| `adminUser.delete` | 刪除後台用戶 |
| `role.manage` | 管理角色 |
| `permission.manage` | 管理權限 |
| `menu.manage` | 管理後台選單 |
| `member.read` | 查看會員 |
| `booking.updateStatus` | 更新預約狀態 |
| `content.publish` | 發布內容 |

### 5.3 預設角色權限

| 功能模組 | 超級管理員 | 管理員 | 內容編輯 | 客服 / 顧問助理 |
|----------|------------|--------|----------|----------------|
| 後台用戶管理 | 全部 | 讀取 | 無 | 無 |
| 角色管理 | 全部 | 讀取 | 無 | 無 |
| 權限管理 | 全部 | 讀取 | 無 | 無 |
| 選單管理 | 全部 | 讀取 | 無 | 無 |
| 會員管理 | 全部 | 全部 | 無 | 讀取、更新備註 |
| 預約管理 | 全部 | 全部 | 讀取 | 讀取、更新狀態 |
| 內容管理 | 全部 | 全部 | 新增、編輯、預覽、送審 | 讀取 |
| 系統設定 | 全部 | 部分 | 無 | 無 |

## 6. CRUD 功能矩陣

### 6.1 通用 CRUD 規格

所有後台管理模組需遵循以下規格：

| 操作 | 規格 |
|------|------|
| Create | 新增資料、欄位驗證、重複檢查、成功 / 失敗提示 |
| Read | 列表、詳細頁、搜尋、篩選、排序、分頁、狀態標籤 |
| Update | 編輯資料、狀態切換、批次更新、異動紀錄 |
| Delete | 軟刪除優先、刪除確認、批次刪除、保留稽核紀錄 |

### 6.2 模組 CRUD 矩陣

| 模組 | Create | Read | Update | Delete | 補充 |
|------|--------|------|--------|--------|------|
| 後台用戶管理 | 新增管理員 | 列表、詳情、篩選角色 | 編輯資料、啟用停用、指派角色 | 軟刪除 | 不可刪除最後一位超級管理員 |
| 角色管理 | 新增角色 | 列表、詳情、權限檢視 | 編輯角色、綁定權限 | 軟刪除 | 系統預設角色不可刪除 |
| 權限管理 | 新增權限 | 權限列表、依模組篩選 | 編輯名稱、描述、群組 | 原則上停用 | 權限代碼建立後不建議修改 |
| 選單管理 | 新增選單節點 | 樹狀列表、選單詳情 | 編輯路由、排序、權限、顯示狀態 | 軟刪除 | 支援多層級與拖曳排序 |
| 會員管理 | 後台新增會員 | 列表、詳情、搜尋、篩選 | 編輯會員資料、狀態、備註 | 軟刪除 | 需保留預約紀錄 |
| 預約管理 | 後台新增預約 | 列表、詳情、日曆檢視 | 改期、取消、確認、完成、備註 | 軟刪除或取消 | 需避免同時段重複預約 |
| 服務項目管理 | 新增服務 | 列表、排序、啟用狀態 | 編輯服務說明、時長、排序 | 停用優先 | 已有預約的服務不可硬刪除 |
| 時段管理 | 新增可預約時段 | 日曆、列表、狀態篩選 | 封鎖、開放、調整時間 | 刪除未預約時段 | 已預約時段不可直接刪除 |
| 文章管理 | 新增文章 | 列表、預覽、依狀態篩選 | 編輯內容、SEO、分類、發布 | 軟刪除 | 對應 `content/blog` |
| 案例管理 | 新增案例 | 列表、詳情、產業篩選 | 編輯案例內容與標籤 | 軟刪除 | 對應 `content/cases` |
| 課程管理 | 新增課程 | 列表、詳情、狀態篩選 | 編輯大綱、適合對象、形式 | 軟刪除 | 對應 `content/courses` |
| 模板管理 | 新增模板 | 列表、詳情、格式篩選 | 編輯說明、下載連結、內容 | 軟刪除 | 對應 `content/templates` |

## 8. 會員管理系統

### 8.1 會員系統目標

會員系統的第一階段目標，是讓使用者能完成以下閉環：

```text
註冊 / 登入 → 補齊會員資料 → 選擇服務與時段 → 送出預約 → 收到確認通知 → 於會員中心查看預約狀態
```

此流程需降低預約前的操作阻力，但仍需確保每筆預約可對應到明確會員身分，方便後續服務追蹤、預約管理與顧問關係經營。

### 8.2 會員註冊與登入

會員端需支援：

- Email / 密碼註冊
- Email / 密碼登入
- Google 帳號登入
- LINE 帳號登入
- 登出
- 忘記密碼
- 重設密碼
- Email 驗證
- 第三方帳號綁定與解除綁定
- 登入後返回原本預約流程
- 已登入會員再次進入登入頁時，自動導向會員中心或原始返回頁

### 8.3 登入與預約導流規則

| 情境 | 系統行為 |
|------|----------|
| 訪客直接進入 `/booking` | 可先瀏覽服務與可預約時段；送出預約前需登入或註冊 |
| 訪客點選預約送出 | 導向登入 / 註冊頁，並保留已選服務、時段與已填資料 |
| 會員登入成功 | 返回原預約流程，繼續確認並送出預約 |
| 新會員註冊成功 | 若需 Email 驗證，先提示驗證；若允許立即預約，需標示帳號待驗證 |
| 會員已登入 | 預約表單自動帶入姓名、Email、電話、公司等會員資料 |
| 會員狀態為停用或黑名單 | 不可建立預約，顯示聯絡管理員提示 |

### 8.4 第三方登入流程

#### Google 登入

```text
1. 使用者點選「使用 Google 登入」
2. 導向 Google OAuth 授權頁
3. 使用者同意授權
4. 系統取得 Google profile 與 email
5. 若 email 已存在，綁定或登入既有會員
6. 若 email 不存在，建立新會員
7. 建立 session 並導向會員中心
```

Google 登入需取得並保存：

| 欄位 | 用途 |
|------|------|
| Google sub | 第三方帳號唯一識別 |
| Email | 會員帳號識別與通知 |
| Email verified | 判斷是否可視為已驗證信箱 |
| Display name | 預填會員姓名 |
| Avatar URL | 會員頭像，選填 |

#### LINE 登入

```text
1. 使用者點選「使用 LINE 登入」
2. 導向 LINE Login 授權頁
3. 使用者同意授權
4. 系統取得 LINE userId、displayName、profile image
5. 若 LINE 帳號已綁定，直接登入
6. 若未綁定且 email 可用，提示綁定或建立會員
7. 建立 session 並導向會員中心
```

LINE 登入需注意：LINE 帳號不一定提供 Email。若未取得 Email，系統需要求會員補填 Email，並完成 Email 驗證或管理端可追蹤的替代驗證流程。

### 8.5 會員狀態

| 狀態 | 說明 |
|------|------|
| 待驗證 | 已註冊但尚未完成 Email 驗證 |
| 啟用 | 可正常使用會員功能 |
| 停用 | 暫停登入與預約功能 |
| 待審核 | 需管理員確認後啟用 |
| 黑名單 | 禁止登入、預約與再次註冊 |
| 已刪除 | 軟刪除，不顯示於一般列表 |

### 8.6 會員資料欄位

| 欄位 | 必填 | 說明 |
|------|------|------|
| `id` | 是 | 會員 ID |
| `email` | 是 | 登入帳號與通知信箱 |
| `emailVerifiedAt` | 否 | Email 驗證時間 |
| `passwordHash` | 否 | Email / 密碼登入使用，第三方登入會員可為空 |
| `name` | 是 | 姓名 |
| `phone` | 否 | 聯絡電話 |
| `company` | 否 | 公司 / 組織 |
| `jobTitle` | 否 | 職稱 |
| `lineUserId` | 否 | LINE 綁定識別，建議放於 OAuth 關聯表 |
| `googleSubject` | 否 | Google 綁定識別，建議放於 OAuth 關聯表 |
| `status` | 是 | 會員狀態 |
| `lastLoginAt` | 否 | 最後登入時間 |
| `createdAt` | 是 | 建立時間 |
| `updatedAt` | 是 | 更新時間 |

### 8.7 會員中心功能

| 功能 | 規格 |
|------|------|
| 個人資料 | 查看與編輯姓名、電話、公司、職稱 |
| 帳號安全 | 修改密碼、查看登入方式、綁定或解除 Google / LINE |
| 我的預約 | 查看預約列表、預約詳情、目前狀態、服務項目、時段與管理端回覆 |
| 取消預約 | 允許在可取消期限內取消，取消後寄送通知 |
| 申請改期 | 會員選擇新時段或填寫改期需求，狀態改為改期中 |
| 通知紀錄 | 顯示重要系統通知，後續擴充 |

### 8.8 後台會員管理

後台需提供會員管理列表與詳情：

| 功能 | 規格 |
|------|------|
| 會員列表 | 搜尋姓名、Email、電話、公司；依狀態、註冊來源、建立時間篩選 |
| 會員詳情 | 顯示基本資料、登入方式、預約紀錄、狀態與管理備註 |
| 狀態管理 | 啟用、停用、待審核、黑名單、解除黑名單 |
| 預約關聯 | 從會員詳情可查看該會員所有預約 |
| 管理備註 | 僅後台可見，用於服務追蹤 |
| 資料保護 | 不顯示密碼雜湊，不提供明文密碼 |

## 9. 預約管理流程

### 9.1 預約流程設計原則

預約流程以會員身分為核心，但不應在一開始阻擋訪客瀏覽服務與時段。建議採用「先選擇、後登入、再確認」的流程，讓使用者先建立預約意圖，再完成身分驗證。

每筆預約需具備：

- 明確會員 ID
- 明確服務項目
- 明確時段
- 預約狀態
- 聯絡資料快照
- 需求說明
- 後台管理備註
- 狀態異動時間

### 9.2 會員端完整流程

```text
1. 選擇服務項目
2. 選擇可預約時段
3. 填寫或確認聯絡資料與需求說明
4. 若尚未登入，導向登入 / 註冊 / Google / LINE 登入
5. 登入成功後返回預約確認頁
6. 檢查時段是否仍可預約
7. 送出預約
8. 系統建立預約並鎖定時段
9. 顯示預約確認頁
10. 發送 Email 通知
11. 會員可於會員中心查看預約狀態
```

### 9.3 會員端例外流程

| 情境 | 系統行為 |
|------|----------|
| 登入後時段已被預約 | 提示時段不可用，返回時段選擇並保留其他表單資料 |
| 會員資料不完整 | 要求補齊姓名與 Email，電話與公司可依服務需求設定為選填或必填 |
| Email 尚未驗證 | 可依系統設定決定是否允許送出預約；若允許，後台需顯示待驗證標記 |
| 預約送出失敗 | 顯示可理解錯誤訊息，不清空使用者已填內容 |
| 會員取消預約 | 更新狀態為已取消，釋放或不釋放時段需依管理規則設定 |
| 會員申請改期 | 原預約標示改期中，新時段需待後台確認 |

### 9.4 管理端流程

```text
1. 後台查看待確認預約
2. 檢視會員資料與預約需求
3. 確認、取消、改期或標記需補充資料
4. 填寫管理員備註
5. 系統發送狀態異動通知
6. 預約完成後標記為已完成或未出席
```

### 9.5 預約狀態

| 狀態 | 說明 |
|------|------|
| 待確認 | 會員已送出，等待管理員確認 |

## 11. UI / UX 規格

### 11.1 整體設計原則

- 符合 AIDC.work 品牌定位：專業、實務、清楚、顧問式
- 不使用過度炫技、過度行銷或干擾操作的視覺效果
- 後台以資訊密度、掃描效率與長時間操作舒適度為優先
- 前台會員流程需降低認知負擔，明確顯示目前步驟與下一步
- 所有表單需提供明確錯誤提示、成功提示、載入狀態與防重送機制

### 11.2 登入頁 UI / UX

登入頁需具備：

- 清楚的 AIDC.work 品牌識別
- Email / 密碼登入表單
- Google 登入按鈕
- LINE 登入按鈕
- 忘記密碼入口
- 註冊入口
- 錯誤訊息與載入狀態
- 行動裝置友善版面

第三方登入按鈕需清楚標示服務來源，例如：

- 使用 Google 登入
- 使用 LINE 登入

### 11.3 後台 UI / UX

後台需符合以下規格：

- 左側固定或可收合功能選單
- 支援多層級選單展開 / 收合
- 頁面標題、麵包屑與主要操作按鈕位置一致
- 列表頁支援搜尋、篩選、排序、分頁與批次操作
- 詳情頁清楚呈現狀態、建立時間、更新時間與操作紀錄
- 表單頁採一致欄位排版與驗證訊息
- 危險操作需二次確認
- 權限不足時顯示明確提示，不暴露不必要資訊

### 11.4 無障礙與響應式

- 支援桌機、平板與手機
- 表單欄位需有 label
- 按鈕文字需明確
- 錯誤訊息不可只依賴顏色
- 鍵盤可操作主要表單與選單
- 色彩對比需符合基本可讀性

## 16. 驗收標準

### 16.1 功能驗收

- 後台用戶可依角色看到不同選單與功能
- 每個 CRUD 模組至少支援新增、列表、詳情、編輯、停用或刪除
- 列表頁具備搜尋、篩選、排序與分頁
- 表單具備欄位驗證、錯誤提示與成功提示
- 會員可註冊、登入、登出與查看個人資料
- 會員登入後可建立預約，預約需正確綁定會員 ID
- 會員可在會員中心查看自己的預約列表與預約詳情
- 會員不可查看其他會員的預約資料
- 預約送出前需再次檢查時段是否可用，避免重複預約
- 預約狀態可由後台正確更新
- 預約狀態更新後，會員端可看到最新狀態
- 內容可依狀態區分草稿、已發布與已下架

### 16.2 UI / UX 驗收

- 登入頁可清楚辨識 Email、Google、LINE 登入方式
- 訪客從預約流程導向登入後，可返回原預約流程
- 會員中心需清楚顯示下一場預約、歷史預約與狀態標籤
- 後台列表、表單與詳情頁視覺一致
- 主要操作按鈕位置一致且容易辨識
- 危險操作有二次確認
- 手機與桌機版面不重疊、不溢出
- 錯誤與成功訊息清楚可理解

### 16.3 技術驗收

- 使用 Next.js App Router、TypeScript、Tailwind CSS
- 不引入不必要依賴
- API 需檢查登入狀態與權限
- 涉及資料異動的操作需有後端驗證
- 會員 API 只能存取目前登入會員自己的資料
- 建立預約需在後端交易中檢查時段可用性並鎖定時段
- OAuth callback 需驗證 state，避免 CSRF
- 密碼需使用安全雜湊，不可明文儲存
- 軟刪除資料不得影響既有預約紀錄完整性

## 17. 風險與注意事項

| 風險 | 說明 | 建議 |
|------|------|------|
| MVP 範圍膨脹 | RBAC、會員、OAuth、CMS 同時實作會提高複雜度 | 依優先順序分階段交付 |
| OAuth 設定成本 | Google 與 LINE 需建立應用程式與 callback URL | 正式版前先確認網域與環境變數 |
| 內容管理轉換成本 | 從 MDX 檔案轉為資料庫管理需調整讀取方式 | 先保留檔案式內容，後續再建立後台 CMS |
| 權限設計過細 | 過度細分權限會增加維護成本 | 初期以模組級權限為主 |
| 刪除資料風險 | 會員、預約與內容資料需可追蹤 | 預設採軟刪除與稽核紀錄 |
| UI 一致性 | 模組增加後容易出現介面不一致 | 建立共用表格、表單、狀態標籤與確認對話框元件 |

## 18. 待確認事項

- 會員是否必須登入後才能預約，或允許訪客預約後再建立會員。
- 建議採「訪客可先選服務與時段，送出前必須登入或註冊」作為第一階段流程。
- Google / LINE 登入是否同時開放於前台會員與後台管理員。
- LINE 未提供 Email 時，是否允許先建立待補資料會員。
- Email 未驗證會員是否可送出預約。
- 會員可取消預約的期限，例如預約前 24 小時或 48 小時。
- 改期流程是否由會員直接選新時段，或先送出改期需求由管理員處理。
- 內容管理正式版要使用資料庫儲存，或維持後台產生 MDX 檔案的模式。
- 會員狀態是否需要人工審核，或 Email 驗證後自動啟用。
- 是否需要完整操作稽核紀錄作為第一階段交付項目。
